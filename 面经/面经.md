# openclaw、herme-agent、自己的区别

这一节用于回答面试官可能追问的开放问题：你怎么看 ByteDance DeerFlow？它和 OpenClaw、Hermes Agent 这类开源 Agent 项目有什么区别？可以用来证明你不是只会讲自己项目，也理解当前 Agent 工程化趋势。

### DeerFlow 的特点

DeerFlow 2.0 的定位是 **Super Agent Harness**，不是单纯聊天机器人，也不是单一 Deep Research Agent。它更像一个 Agent 运行底座，把模型、工具、技能、文件系统、沙箱、长期记忆、子 Agent 编排、Web UI 和 Gateway API 组合在一起，让 Agent 能执行较长时间、较复杂的任务。

它的核心特点主要有 7 个：

2. **Lead Agent + Sub-Agents 架构**
  DeerFlow 有一个主 Agent 负责规划、拆解和综合，复杂任务可以动态创建多个子 Agent。每个子 Agent 有独立上下文、工具和终止条件，可以并行执行，最后把结构化结果交回主 Agent 汇总。

3. **Skills 驱动能力扩展**

  DeerFlow 通过 Skill 扩展能力。一个 Skill 通常由 `SKILL.md` 描述工作流、最佳实践、可用工具和参考资源。它的优势是按需加载技能，而不是一开始把所有能力都塞进上下文，适合长任务和多能力组合。

4. **沙箱和文件系统是核心能力**

  每个 thread 有独立的 `uploads`、`workspace`、`outputs`。Agent 可以读写文件、生成交付物、执行命令。生产环境推荐 Docker sandbox，本地模式下 host bash 默认不应开放，因为本地执行不是强安全隔离边界。

5. **重视 Context Engineering**

  DeerFlow 会隔离子 Agent 上下文、总结已完成子任务、把中间结果下沉到文件系统，并压缩不再关键的信息。它解决的是长任务中上下文爆炸、状态混乱和中间产物管理的问题。

6. **兼容 LangGraph / LangChain 生态**

  DeerFlow 后端 Agent runtime 基于 LangGraph，模型调用和工具抽象基于 LangChain。对于已经使用 LangGraph 的团队来说，它不是一个完全陌生的 Agent 框架，而是更像工程化包装和应用化扩展。


### DeerFlow 的设计理念

DeerFlow 的设计理念可以概括成一句话：

> 给 Agent 一套可靠的工作环境，而不是只给模型一堆工具。

它强调的不是单纯让模型更聪明，而是给 Agent 配齐真实完成任务所需的运行时基础设施：

- 可扩展的 Skill 体系
- 真实文件系统和产物目录
- 可控沙箱
- 长任务上下文治理
- 主 Agent 和子 Agent 协作
- 长期记忆
- Web/Gateway/API 部署形态
- MCP 和外部工具接入能力

所以 DeerFlow 的工程气质是 **平台化、工作流化、可部署化**。它关心的问题是：一个 Agent 要长期、稳定、可控地完成复杂任务，除了 LLM 本身，还需要哪些运行时能力。

### 和 OpenClaw 的区别

OpenClaw 更像 **local-first personal AI assistant**，也就是运行在用户自己设备上的个人 AI 助手。它强调通过 WhatsApp、Telegram、Slack、Discord、iMessage、微信、QQ 等消息渠道和用户交互，也支持 CLI、桌面端、移动端 companion app、语音和 Live Canvas。

| 维度 | DeerFlow | OpenClaw |
|---|---|---|
| 核心定位 | Super Agent Harness / 长任务工作台 | Local-first personal AI assistant |
| 主要入口 | Web UI、Gateway API、IM 渠道、Python client | 消息渠道、CLI、桌面/移动 companion app |
| 重点 | 深度任务、产物生成、多 Agent、沙箱、上下文工程 | 个人助理、常驻、跨 IM 渠道、设备和应用控制 |
| 架构气质 | LangGraph + FastAPI + Next.js，偏工程平台 | Gateway daemon + 多渠道连接，偏个人助手系统 |
| 多 Agent | 主 Agent 动态拆解子 Agent | 支持多 Agent 路由到隔离 workspace/session |
| 安全模型 | 线程级文件区，生产推荐 Docker sandbox | main session 更偏本机助手，其他 session 可 sandbox |
| 典型任务 | 调研报告、数据整理、网页/幻灯片/文件产物 | 收消息、调度、操作本地工具、跨平台个人自动化 |

一句话区别：

> OpenClaw 更像 AI 住进你的电脑和聊天软件里；DeerFlow 更像给 AI 一个可以完成复杂任务的工作台。

### 和 Hermes Agent 的区别

Hermes Agent 的定位更偏 **self-improving AI agent**，也就是自我改进型 Agent。它特别强调从经验中创建技能、在使用中改进技能、主动沉淀记忆、搜索过去会话、逐渐建立用户模型。

| 维度 | DeerFlow | Hermes Agent |
|---|---|---|
| 核心定位 | 通用 Super Agent Harness | 自我学习、自我改进 Agent |
| 最强卖点 | 多 Agent + 沙箱 + 产物工作流 + Web/Gateway | 学习循环、记忆、技能自生成/自改进 |
| 主要入口 | Web UI、API、IM、Python client | TUI/CLI、Messaging Gateway |
| 运行环境 | 本地、Docker、Kubernetes sandbox，偏服务化部署 | 本地、Docker、SSH、Singularity、Modal、Daytona、Vercel Sandbox 等 |
| 记忆 | 长期记忆，保存用户偏好和项目知识 | 更强调 agent-curated memory、会话搜索、用户建模 |
| 技能 | 按需加载、可扩展、可组合 | 可从复杂任务中自动创建并改进技能 |
| 典型任务 | 研究、报告、网页、幻灯片、文件处理 | 长期个人助手、自动化、周期任务、经验积累 |

一句话区别：

> Hermes 更关心 Agent 会不会越用越懂你、越用越会做事；DeerFlow 更关心 Agent 有没有一套稳定运行时去完成复杂任务。

### 三者总结

- **DeerFlow**：面向复杂任务和交付物的 Agent 工作台 / runtime。
- **OpenClaw**：面向个人日常自动化的本地常驻助手。
- **Hermes Agent**：面向长期陪伴、记忆沉淀和自我学习的 Agent。

如果目标是研究报告生成、企业内部 Agent 平台、多 Agent 工作流、文件产物生成，DeerFlow 更合适。

如果目标是接入微信、Slack、Telegram，让 AI 常驻在个人设备和聊天渠道里，OpenClaw 更贴近。

如果目标是研究 Agent 如何积累经验、自动沉淀技能、建立长期记忆和自我改进，Hermes Agent 更有代表性。

### 面试回答模板

如果面试官问“DeerFlow 有什么特点？和 OpenClaw、Hermes Agent 有什么区别？”，可以这样回答：

> 我理解 DeerFlow 2.0 的核心不是做一个聊天机器人，而是做一个 Super Agent Harness。它把模型、工具、Skill、文件系统、沙箱、长期记忆、子 Agent、Web UI 和 Gateway 组织成一个完整运行时，让 Agent 能执行长任务并产出真实文件。它的设计理念是给 Agent 一个可靠工作环境，而不是只给模型一堆工具。
>
> 和 OpenClaw 相比，OpenClaw 更像 local-first personal assistant，重点是接入各种 IM 和本机设备，成为个人常驻助手；DeerFlow 更像任务工作台，重点是复杂任务、产物生成、多 Agent 和 sandbox。
>
> 和 Hermes Agent 相比，Hermes 更强调 self-improving，也就是 Agent 从历史任务里自动沉淀记忆和技能，越用越会做事；DeerFlow 更强调运行时工程化，比如上下文治理、文件系统、沙箱、Gateway、Web UI 和多 Agent 编排。
>
> 所以如果我要做企业内部复杂任务 Agent 或研究报告/文件产物生成，我会优先看 DeerFlow；如果我要做个人设备上的常驻助手，我会看 OpenClaw；如果我要研究长期记忆和自我改进机制，我会看 Hermes Agent。


# deerflow是如何确保 复杂任务和交付物的？具体是怎么实现的？

## 1，Agent runtime 架构：
```
主 Agent 负责规划与协调，工具系统负责真实执行，沙箱负责文件和命令环境，Skill 负责领域工作流，Subagent 负责并行拆解，Memory/ThreadState/Checkpointer 负责状态延续，outputs/artifact API 负责把最终产物交给用户。
```

简单LLM调用：```用户输入 -> LLM -> 文本回答```
deer_flow:
```
用户输入
-> Gateway / LangGraph run
-> make_lead_agent(config)
-> 加载模型、工具、Skill、Memory、ThreadState
-> LangGraph Agent 循环执行
-> 工具调用 / 文件读写 / 子 Agent / 沙箱执行
-> SSE 流式返回
-> 输出文件作为 artifact 暴露给用户
```

## 2，复杂任务靠 Lead Agent + Subagent 拆解

DeerFlow 的复杂任务不是让一个 Agent 从头做到尾，而是让 lead agent 在需要时调用 task 工具，把子任务交给 subagent。

执行流程：
```
lead agent 调用 task()
-> SubagentExecutor 创建后台任务
-> 子 Agent 在独立执行上下文里运行
-> 后台轮询状态
-> 通过 SSE 发送 task_started / task_running / task_completed 等事件
-> 子 Agent 结果返回 lead agent
-> lead agent 汇总
```
从代码设计角度看，这解决了两个问题：
- 任务拆解：复杂任务可以分成研究、编码、文件处理、验证等多个子任务。
- 上下文隔离：子 Agent 有自己的执行上下文，不会把所有中间过程都塞进主 Agent 上下文。

 ReAct Agent 最大区别是：普通 Agent 往往是一条线反复思考和调用工具；DeerFlow 更像一个调度器，可以把任务拆出去并收敛结果。


## 3，交付物靠 Thread 文件系统，而不是只靠文本回复

DeerFlow 对“交付物”的支持，核心在 thread 级文件系统。
每个 thread 都有独立目录：
```
/mnt/user-data/workspace
/mnt/user-data/uploads
/mnt/user-data/outputs
```
这些虚拟路径会映射到真实物理路径。这就是它能生成报告、网页、幻灯片、代码文件的底层原因：**输出被设计成文件产物，而不是聊天消息的副产品。**

## 4. 沙箱让工具执行变成可控能力

沙箱解决的是：
```text
LLM 想做事
-> 不能直接碰宿主机
-> 只能通过受控 sandbox 工具
-> 文件路径限制在 thread workspace / uploads / outputs
-> 命令执行也受 sandbox provider 管理
```

所以 DeerFlow 的“能交付”不是只靠模型会写代码，而是因为它真的给 Agent 提供了一个可执行环境。



# 介绍一下这个项目
```
这个项目是我基于 DeerFlow / LangGraph 做的一套自定义 Agent 运行框架。它的核心不是简单接一个 LLM 聊天，而是把 Agent 的身份、会话、记忆、工具和文件工作区拆成了清晰的边界。

在系统里，agent_name 表示当前使用哪个 Agent，它对应独立的 config.yaml、SOUL.md 和 memory.json。SOUL.md 定义 Agent 的角色、人设、沟通方式和行为边界；memory.json 存长期记忆。thread_id 则表示一次会话线程，它负责 LangGraph checkpoint、消息状态和独立 workspace。这样不同 Agent 的人格和记忆不会混在一起，不同会话的文件也不会互相覆盖。

运行时，系统会根据当前 agent_name 加载配置、模型、工具组、SOUL 和 memory，然后通过 middleware 组合出完整 Agent。记忆更新不是实时写日志，而是在主 Agent 完成一轮回复后，由 MemoryMiddleware 过滤掉工具调用和上传文件信息，再放进带 debounce 的后台队列，由 LLM 结构化更新长期记忆。

工具方面，我做了工具组过滤、MCP 工具缓存和延迟暴露，并通过 sandbox 路径映射把 Agent 的文件访问限制在 /mnt/user-data/workspace、uploads、outputs 这些虚拟路径里，真实落盘到当前 thread_id 对应的目录，从而保证文件读写隔离和安全。

所以这个项目的重点是：自定义 Agent 生命周期、长期记忆沉淀、会话隔离、工具治理和 sandbox 文件边界。
```

# 这个项目有什么难点？
```
我觉得主要有四个难点：
第一个是状态边界设计。Agent 系统里有多种状态，不能混在一起。我把 agent_name 和 thread_id 分开：agent_name 管 Agent 的配置、人设和长期记忆，thread_id 管会话 checkpoint 和 workspace。这个边界是后面所有隔离能力的基础。
第二个是记忆污染问题。长期记忆不能等同于聊天日志。工具调用、中间推理、上传文件路径这些东西都不应该被写进 memory。所以我在 MemoryMiddleware 里只保留 human message 和最终 AI response，并且移除 <uploaded_files>，再通过后台队列和 LLM 结构化更新 memory。

第三个是工具规模和可控性。MCP 工具很多，全部暴露会造成 token 压力，也会让模型选择困难。因此我做了工具 group 过滤和延迟工具加载。模型一开始只看到 deferred tool 的名称，需要时再通过 tool_search promote 出具体工具 schema。

第四个是文件访问安全。Agent 能读写文件和执行命令，这个能力很强，但风险也高。我用 thread 级 workspace 做隔离，把 Agent 可见路径限制在 /mnt/user-data/workspace、uploads、outputs，真实路径由系统映射，同时拒绝 .. 路径穿越和不安全绝对路径。

```

# 如果你的agent项目要正式部署给企业使用，需要优化哪些方面? 

```
第一是并发和可扩展性。现在项目更偏单机 Agent Runtime，企业级需要 API 网关、异步任务队列、worker 扩缩容、分布式 checkpointer、模型调用并发控制和限流。
第二是多租户和权限隔离。现在我做的是 agent_name 和 thread_id 级隔离，但企业里还需要 tenant、user、role 级别的权限模型，确保不同部门的数据、memory、workspace 和工具权限不会串。
第三是安全治理。Agent 有读写文件和工具调用能力，企业里必须加强 sandbox 隔离、host bash 禁用、工具白名单、secret 脱敏、上传文件扫描、外部访问白名单和审计。
第四是记忆治理。当前记忆更新是主 Agent 结束后异步沉淀，但企业环境里还需要 PII 检测、敏感信息过滤、记忆可查看、可删除、可回滚和生命周期策略。
第五是可观测性和稳定性。需要 trace、日志、token 成本统计、工具调用审计、失败率监控、告警，以及模型和工具失败后的重试、降级、熔断和补偿机制。
所以如果让我继续做，我会先把它从本地 runtime 改造成服务化架构，再补齐租户权限、安全审计、任务队列和可观测性。高并发只是其中一块，而且这块我目前没有深入实现，比如分布式队列、K8s 扩缩容和服务治理还没有覆盖。
```

# llm返回格式是什么？怎么解析出tool工具结果？

## 知识点：

- 第一层：LLM 原始返回，本质上是结构化 JSON / message 对象
- 第二层：项目里不直接靠 re 解析工具调用，而是交给 LangChain/LangGraph 抽象成 AIMessage.tool_calls 和 ToolMessage。

### 第一层，底层逻辑
普通文本回复：```{"role": "assistant", "content": "这是最终回答。"}```
工具调用回复：
```
{
  "role": "assistant",
  "content": "",
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "read_file",
        "arguments": "{\"path\":\"/mnt/user-data/workspace/a.txt\"}"
      }
    }，
    {}
  ]
}
```
工具返回可能有规范的样式，tool_calls；还有的模型不支持原生 function calling，而是把 JSON 混在普通文本里，这个时候就需要`re` 或 `JSONDecoder` 从文本里提取 JSON。但是，**最好走API 的 `tool_calls` 结构，而不是从自然语言里抠 JSON**

大致逻辑：
```
1. 用 re 去掉 <think>...</think>
2. 去掉 ```json 代码块
3. 尝试 json.loads()
4. 如果失败，就从文本中找到第一个 {，用 JSONDecoder.raw_decode() 尝试解析
```





### 第二层，工具调用

当模型想调用工具时，会产生：
```
AIMessage(
    content="",
    tool_calls=[
        {
            "name": "read_file",
            "args": {"path": "/mnt/user-data/workspace/a.txt"},
            "id": "call_abc123"
        }
    ]
)
```

如果AIMessage中存在tool_calls，agent graph 进入工具执行逻辑     —— 根据 name 找到对应工具，用 args 调用工具；执行完后会产生ToolMessage，
```
ToolMessage(
    content="文件内容...",
    name="read_file",
    tool_call_id="call_abc123"
)
```
关键是，```AIMessage.tool_calls[i].id == ToolMessage.tool_call_id```;

如果工具失败怎么办

工具失败也会变成 ToolMessage，只是状态是 error。
```python
return ToolMessage(
    content="Tool call failed: ...",
    tool_call_id=tool_call_id,
    name=tool_name,
    status="error",
)
```

如果 AIMessage 有 tool_calls，但没有 ToolMessage 怎么办?
比如：
```
AIMessage 里有 tool_calls
但后面没有对应的 ToolMessage
例子：比如用户中断、请求取消、工具没返回。它会补一个合成的错误 ToolMessage：
```

```
ToolMessage(
    content="[Tool call was interrupted and did not return a result.]",
    tool_call_id=tc_id,
    name=tc.get("name", "unknown"),
    status="error",
)
```

## 面试版本
```
LLM 返回我分两类处理。第一类是普通结构化 JSON，比如记忆更新这种场景，我会在 prompt 里要求模型只返回 JSON，然后在后处理里做容错解析：先用正则去掉 <think> 和 markdown code block，再用 json.loads 解析，如果失败就用 JSONDecoder.raw_decode 从文本里提取第一个 JSON 对象。这样可以应对模型偶尔包一层说明文字或者代码块的问题。

第二类是工具调用。工具调用我没有用正则从自然语言里解析，因为那不稳定。我是依赖模型原生 function calling，再由 LangChain 统一抽象成 AIMessage.tool_calls。每个 tool call 里有 name、args 和 id。LangGraph 根据 name 找到对应工具，把 args 传进去执行。工具执行结果会被包装成 ToolMessage，里面有 content、name 和 tool_call_id。这个 tool_call_id 和前面 tool call 的 id 对应，所以框架能知道这个工具结果属于哪一次调用。

在我的项目里，流式输出时会分别处理三种消息：如果是 AIMessage.content，就作为普通 AI 文本输出；如果是 AIMessage.tool_calls，就输出工具调用事件；如果是 ToolMessage，就输出工具结果事件。工具失败也不会直接抛给用户，而是通过 middleware 转成 error 类型的 ToolMessage，让模型下一轮可以继续基于错误信息处理。
```

# 如果要设计权限，每个人对于每个文档权限改怎么设计?实际一些

## 知识点
每个人对每个文档的权限，拆为三层
```text
用户是谁：user
文档属于哪里：document / workspace / tenant
用户能做什么：permission
```

最小可用模型：
```text
tenant      企业 / 组织
workspace   项目空间 / 知识库空间
document    文档
user        用户
role        角色
permission  具体动作
```

文档权限：不要只设计成一个 can_read=true/false，最好拆成动作：
```text
read      查看
comment   评论
edit      编辑
share     分享
delete    删除
admin     管理权限
```

分组也很重要：因为实际企业里不会只给单个人授权，还会给部门、用户组、角色授权。
```text
user: 张三 对 documentA 有 edit
group: 算法组 对 documentA 有 read
role: workspace_admin 对 documentA 有 admin
```

**Agent 场景下还要额外注意**

因为 Agent 会调用 ```read_file```、知识库检索、MCP 工具。

```
Agent 调 document_search
  -> 带上 user_id / tenant_id / thread_id
  -> 检索时只查用户有 read 权限的文档
  -> 返回结果前再次过滤 document_id

Agent 读某个文档
read_document(document_id)
  -> check_permission(user_id, document_id, "read")
  -> allowed 才返回内容

Agent 读多个文档
summarize_documents([doc1, doc2, doc3])
  -> doc1 allowed
  -> doc2 denied
  -> doc3 allowed
  -> 只处理 allowed docs
  -> 告诉用户 doc2 没权限
```

文档内容如果进入长期记忆，可能造成权限泄露。所以企业版要加：
```
memory 写入前做权限和敏感信息过滤
memory 按 user_id / tenant_id / agent_id 隔离
不要把受权限控制的文档原文沉淀到共享 memory
```


## 面试

```
如果要设计每个人对每个文档的权限，我不会只做一个简单的 user_id-document_id 映射，而是会设计成用户、文档、空间和权限动作四层。文档属于某个 tenant 和 workspace，用户也属于 tenant，可以通过个人、用户组、workspace role 获得权限。权限动作会拆成 read、comment、edit、share、delete、admin，而不是只有一个 can_read。

表上我会有 documents、users、groups、group_members、workspace_members 和 document_permissions。document_permissions 里用 subject_type + subject_id 表示授权对象，可以是 user、group 或 role，然后 permission 表示具体动作。

请求进来时，先从 token 拿 user_id 和 tenant_id，再查文档，第一步校验 tenant_id，防止跨租户访问；然后判断 owner、workspace 成员、可见性和显式授权。只有权限通过，才允许读取文档内容。如果文档在对象存储里，也不会直接暴露真实 URL，而是后端校验通过后生成短期 signed URL。

如果放到 Agent 场景，还要在工具层做权限校验。比如 document_search、read_document、summarize_documents 这些工具都必须带 user_id 和 tenant_id，只能检索和读取用户有 read 权限的文档。不能只靠前端隐藏按钮，因为 Agent 可能绕过 UI 直接调用工具。同时还要避免把无权限文档内容写进共享 memory，防止后续对话泄露。
```

# 如果是在RAG中，怎么控制对所有文档中某些文档的权限？

最实用的做法不是只靠前端，而是把权限压在检索前、检索中、返回前三道关

1. 文档入库时就带权限元数据
每个 chunk 都要存这些字段：

```
{
  "doc_id": "d123",
  "tenant_id": "t1",
  "workspace_id": "w9",
  "owner_id": "u7",
  "acl": ["u7", "group:legal", "role:manager"],
  "visibility": "private|workspace|tenant"
}
```

2. 检索时先做权限过滤，再搜向量
先拿到当前用户的可见文档集合，再去向量库里搜。
这是最稳的，避免“先搜出来再删”，因为后者容易泄露相似度和摘要。

3. 如果向量库支持 metadata filter，就直接过滤

4. 返回结果前再做一次权限校验
防止检索链路里有漏网之鱼。
这步尤其重要，因为 RAG 经常会把多个来源拼起来。

## 两种工程方案
A: 单库 + metadata filter
复杂权限都放到 metadata 里

B: 按 tenant / workspace 分库

企业里通常会混着用：
tenant 级别分库
workspace/doc 级别再用 metadata ACL

实际请求流程:
```
用户请求
  -> 认证拿到 user_id / tenant_id / role
  -> 查用户能访问的 workspace/doc ACL
  -> 过滤 chunk 集合
  -> 只对允许的 chunk 做 embedding search
  -> 召回结果再校验一次
  -> 拼上下文给 LLM
```

## 面试

```
RAG 的权限控制，我会放在检索链路里做，不是只靠前端。文档入库时会带 tenant、workspace、owner 和 ACL 这些元数据，检索前先根据当前用户的身份过滤可见文档，再去向量库做召回；如果向量库支持 metadata filter，就直接在查询里做权限条件。召回后还会再做一次权限校验，避免跨租户或者跨 workspace 泄露。
如果是企业级场景，我会优先做 tenant 级隔离，再在 tenant 内用 workspace 和 doc ACL 做细粒度控制。对于 Agent 场景，权限还要下沉到 search/read/summarize 工具层，不能只隐藏 UI，否则 Agent 可能绕过界面直接读文档。
```
