# DeferredToolRegistry机制，跟SKILL渐进式披露不是一样的吗？都是给name和description，为什么不用SKILL来代替？

相似，但不是同一层东西；
```text
一句话
它们长得像，都是渐进式披露；但 Skill 披露的是“怎么做”，DeferredToolRegistry 披露的是“能调用什么以及怎么调用”。MCP 工具必须进入 LangChain 的工具绑定和 ToolNode 执行链路，所以不能只用 Skill 替代。
```

## DeferredToolRegistry 是执行层机制
MCP 工具最终要被模型调用，模型必须看到完整 JSON schema。否则它不知道参数名、类型、必填项，也无法稳定生成合法的 tool call。
流程是：
```text
system_prompt 里列 deferred tool name
  -> 模型调用 tool_search
  -> tool_search 返回 OpenAI function schema
  -> registry.promote()
  -> 下一轮 bind_tools 时该工具 schema 真正交给模型
  -> 模型才能调用该工具
  -> ToolNode 执行

管的是：
这个工具现在是否允许进入 request.tools
这个工具 schema 是否要传给模型
这个工具是否可被 ToolNode 执行
```

## SKILL 是提示层机制

Skill 不是可执行工具。它只是
```text
name
description
SKILL.md 路径
references/templates/scripts/assets 路径
```

但 skill 本身不会让模型获得新的 tool-call 能力。它不能替代工具 schema，也不能告诉 LangChain ToolNode 如何执行某个 MCP 工具。

## 为什么不能用 Skill 替代 DeferredToolRegistry

因为SKILL是说明书，并不能给Schema；
问题是：模型看到这些说明后，仍然不能直接调用 github_create_issue，因为这个工具没有进入 request.tools 的 bind schema。OpenAI-compatible 模型的工具调用必须走：

也就是必须把工具 schema 作为 tools 参数传给模型，而不是只写在 system prompt 里。
写在 Skill 里只能影响自然语言推理，不能注册成正式工具。

# 这个项目的工具是用列表存储的，那万一有成千上万个工具，你要怎么办？


所以答案是：不用担心“列表存储”本身，应该担心“工具 schema 暴露策略”。这个项目已经用 deferred tools 做了雏形；如果规模到几千，需要把 DeferredToolRegistry 从内存 list + regex 升级成工具目录索引 + top-k 检索 + schema 懒加载。

应该从以下几个点进行改进：
```
1. DeferredToolRegistry 不再保存完整 BaseTool，只保存 metadata + resolver
2. tool_search 从 regex list 扫描改成 SQLite FTS/BM25/向量检索
3. get_available_tools 不再返回全部 MCP tools，只返回普通核心工具 + tool_search
4. ToolNode 执行 deferred 工具时通过 wrap_tool_call 动态 resolve 工具
5. MCP client 按 server 懒初始化，不在启动时连接所有 server
```


现有的工具占用情况：
```
tools_total:53
工具列表总内存：178.98 KB
name='web_search' description='Search the web.' args_schema=<class 'langchain_core.utils.pydantic.web_search'> func=<function web_search_tool at 0x0000021CF69A1F80>
一个大概是：3.3KB

粗略估计：
1,000 个工具  ≈ 3.3 MB
10,000 个工具 ≈ 33 MB
50,000 个工具 ≈ 169 MB

实际 MCP 工具可能还挂着 coroutine、client/session、args_schema、闭包引用，可能更大。但即使 10,000 个工具，几十到几百 MB 级别在服务端不一定立刻崩。
```

真正危险的是：
```
每轮把大量 schema bind 给模型
大量 Pydantic schema 转 JSON
请求体巨大
token 成本巨大
模型工具选择变差
ToolNode / registry 初始化变慢
MCP server 连接和工具发现变慢
```

当前设计：
```
普通工具：直接暴露
MCP 工具：注册到 DeferredToolRegistry
system_prompt 只列 deferred tool name
模型先调用 tool_search
命中后 promote
下一轮才把少量工具 schema 暴露给模型
```

实际情况：
```
1. 工具元数据索引
   只保存 name、description、server、tags、embedding、schema_ref

2. 工具搜索服务
   用 BM25 / embedding / hybrid search 查 top_k

3. 按需加载 schema
   只把 top 5-20 个工具 schema 放进 request.tools

4. ToolNode 执行层仍持有或可懒加载完整工具对象
   模型可见集合 != 后端可执行集合

5. 分命名空间
   server/tool，例如 github.search_repos、leetcode.get_problem

6. 权限过滤前置
   根据 agent、tool_group、用户权限、sandbox 环境先裁掉不可用工具
```

现有的注册表仍然持有完整 tool 对象。如果工具数量非常大
```
DeferredToolEntry(
    name=tool.name,
    description=tool.description,
    tool=tool,
)

@dataclass
class ToolMetadata:
    name: str
    description: str
    server: str
    tags: list[str]
    schema_ref: str
```

只有搜索命中才获取BaseTool：

```python
resolve(name) -> 加载/获取 BaseTool
convert_to_openai_function(tool) -> 返回 schema
```

tool_search 也不应该再用简单 regex，而应该变成真正的检索：

query -> lexical search + embedding search -> rerank -> return top_k schemas

# MCP会有很多工具，怎么确保agent能准确找到需要的那个？找失败了怎么办

没法“确保”100% 找准，只能通过检索设计和失败恢复把准确率做高。这个项目当前机制是基础版：tool_search 在 DeferredToolRegistry 里按 name + description 做 regex/关键词匹配，最多返回 5 个工具。

支持三种匹配：
```json
select:name1,name2     精确选择
+keyword rest          工具名必须包含 keyword，再排序
普通 query             regex 搜 name + description

例子：
select:read_file, search_code

+file read write
- required = "file" → 候选为 read_file, write_file, delete_file（名中均包含 "file"）
- 剩余部分 "read write" 用于排序：_regex_score("read write", entry) 对这三个条目计算相关性分数，可能 read_file 和 write_file 得分更高，排在前面
- 返回排序后的工具列表（最多 MAX_RESULTS 个）

weather
正则化表达式

```

如果没找到：
```
return f"No tools found matching: {query}"
```
## 当前
主要靠 MCP 工具本身的 ***description***

如果只靠自然语言关键词，比如 "leetcode problem"，当前 regex 搜索就比较粗糙，容易漏掉或返回不相关工具。

## 查找失败
1. tool_search 没搜到。
   模型调用 tool_search
  -> 返回 "No tools found matching: xxx"
  -> 模型可以换 query 再试
  -> 仍失败则说明无法找到相关工具

2. 模型直接调用了还没 promote 的 deferred 工具。
   中间件会进行拦截：
   Error: Tool 'xxx' is deferred and has not been promoted yet. Call tool_search first...

   也就是不会直接执行，而是给模型一个错误 ToolMessage，让它先调用 tool_search
## 提高准确率
如果 MCP 工具很多，建议把当前 regex search 升级成真正的工具检索系统
```text
1. 工具命名规范化
   server.tool_action，例如 leetcode.get_problem、github.search_repos
2. description 写清楚触发场景
   不只写 “Get problem”，要写 “Fetch LeetCode problem by titleSlug/id from leetcode.cn”
3. 增加 tags / aliases
   leetcode, problem, solution, submit, contest
4. tool_search 返回候选工具时带原因
   告诉模型为什么这些工具匹配
5. 支持多轮检索
   第一次搜 broad query，第二次 select 精确工具
6. 用 BM25 / SQLite FTS / embedding hybrid search
   不要只靠 regex
7. 返回 top-k schema，但不要太多
   通常 3-8 个比较合适
```

找不到时不要让模型瞎猜。应该要求它：
```
1. 换关键词再搜一次
2. 如果看到 available-deferred-tools 里有疑似工具，用 select:<exact_name>
3. 如果仍失败，改用普通工具或向用户说明缺少对应 MCP 工具
```
