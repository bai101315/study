## 通用方法论


### 1，规划
目标 —— 拿到技术方案
```
我需要在两个小时内完成以下项目：
[题目的README]
先不要写代码，帮我分析：
1，技术选型建议(给2-3个方案，标注推荐)
2，模块划分和文件结构
3，核心数据模型设计
4，接口/API实现
5，实现优先级排序(哪里是必须做，哪里是加分项)
```

### 2，架构实现
目标：核心后端和数据层跑起来; 能跑起来就继续

```
技术栈确认：[你选的技术栈]
请实现核心代码，包括：
1，[数据模型要求]
2，[业务逻辑要求]
3，[API/接口要求]

验收标准：[写清楚怎么算好了]
```

### 3，前端页面实现
目标：前端实现，一个模块一个模块生成

```
后端API已就绪，现在实现[模块名]：
- [功能点1]
- [功能点2]
- [功能点3]
- 输入非常内容不崩溃
```

### 4，Stabilize稳定化
目标：全流程跑通+测试覆盖

```
项目基本功能完成，请创建[测试文件]，进行测试，覆盖：
- [正向流程测试]
- [异常输入测试]
- [权限/边界测试]

要求：
- 测试自包含，不需要手动启动服务
- 每个测试独立，不依赖执行顺序
- 附上运行命令
```

### 5，交付
目标：补全所有交付物

```
项目基本功能完成，请创建 [测试文件名]（用[测试框架]），覆盖：
- [正向流程测试]
- [异常输入测试]
- [权限/边界测试]

要求：
- 测试自包含，不需要手动启动服务
- 每个测试独立，不依赖执行顺序
- 附上运行命令

同时生成 run_tests.sh：一键运行所有测试并保存报告。
```

## Prompt的四个黄金法则法则
1：给验收标准，不给模糊需求
```
❌ "实现用户管理功能"
✅ "实现用户CRUD API，验收标准：curl能创建用户，能按ID查询，能更新密码"
```

法则2：出错时给完整上下文
```
❌ "报错了，帮我修"
✅ "运行 python server.py 报错：
    [贴完整错误，不要截图]
    相关代码在 server.py 第30行：
    [贴相关代码]
    只修改有问题的部分，不要重写整个文件。"

```

法则3：用选择题代替开放题
```
❌ "你觉得用什么技术？"
✅ "WebSocket和HTTP轮询二选一，考虑2小时时限和终端场景，推荐哪个？"
```


法则4：复杂功能分步确认
```
❌ "实现完整的订单系统"
✅ "先只实现数据模型和内存存储，持久化后面再加。"
```
