# 可插拔骨架理解

```
我这条主要想表达的是运行时解耦。LangGraph 给了我 Agent 和 Graph 的基础执行框架，但真实 Agent 系统还需要很多横切能力，比如工具治理、沙箱、记忆、上下文压缩、错误处理等。我没有把这些逻辑全部写进主 Agent，而是抽象成 Middleware、Tool、Sandbox、Memory 几类接口。

Middleware 负责执行链路治理，Tool 负责统一不同来源的工具，Sandbox 负责本地执行和路径权限，Memory 负责长期记忆沉淀。这样 Agent 主流程只做编排，各能力模块可以按配置启停和替换。比如 tool_search 开启时会插入 DeferredToolFilterMiddleware，关闭时就不影响其他工具；Memory 也是通过 MemoryMiddleware 异步写入，不阻塞主对话。

这个设计解决的问题是：如果所有能力都耦合在主流程里，代码很难维护，也很难在不同 Agent 间复用。抽象成可插拔运行骨架后，新增能力只需要接入对应接口，运行时按配置组装即可。
```

# 项目介绍
```
AgentFlow 是一个基于 LangGraph / LangChain 的本地多 Agent 运行时框架，支持自定义 Agent、工具权限控制、MCP 工具接入、长期记忆、会话持久化、本地沙箱、技能注入和中间件治理。它主要解决的是：当 Agent 从“单轮问答”变成“能使用工具、读写文件、长期记忆、多 Agent 协作”的系统后，如何保证上下文不污染、工具不失控、会话可恢复、记忆可沉淀、执行环境有边界。
```

# 具体解决了什么问题

## 多 Agent 的上下文隔离
普通聊天系统通常只有一个角色和一条会话历史，但这个项目支持创建多个自定义 Agent。每个 Agent 可以有自己的 SOUL.md、模型配置、工具组、技能配置和长期记忆。这样用户可以创建不同用途的 Agent，比如刷题助手、日报助手、代码助手，而不是让所有任务共享同一套系统提示词和记忆。

这里的关键点是：agent_name 负责 Agent 身份、配置和长期记忆；thread_id 负责会话状态和工作目录。也就是说，不同 Agent 的人设、memory、workspace 不会混在一起。

## 工具数量变多后 prompt token 成本和工具选择混乱

项目接入了 MCP 工具，比如 GitHub、LeetCode 等。如果把所有 MCP 工具的完整 schema 都直接暴露给模型，每轮 prompt 都会带很多 function schema，token 成本高，模型选择工具时也容易被干扰。

所以项目实现了 tool_search 延迟工具发现机制。MCP 工具仍然进入执行层，保证真正调用时 ToolNode 能找到它们；但模型初始只在 prompt 里看到工具名，不看到完整 schema。模型需要某个工具时，先调用 tool_search 搜索，系统返回匹配工具的完整 schema，并将该工具 promote，下一轮模型才能真正调用。

这个优化我还做了离线 benchmark：当前配置加载 42 个 MCP tools，全部 schema 初始暴露约 6242 tokens；开启 deferred tool search 后，初始只需要约 469 tokens，工具 schema 相关开销下降约 92.49%。这个数据是 tokenizer 离线估算，不是线上 billing 数据，所以表达时要说“prompt schema overhead 的离线测算”。

## 长期记忆和会话状态的边界。

Checkpointer 保存当前 thread 的 LangGraph 会话状态，比如消息历史、工具调用状态、graph 执行状态。它解决的是“这轮会话怎么恢复、下一轮怎么接着上轮继续”的问题。

memory.json 保存长期记忆，比如用户偏好、项目背景、稳定事实、长期目标。它解决的是“跨会话如何记住有价值的信息”的问题。

这两者不能混用。只用 Checkpointer，历史会越来越长，token 成本高，而且里面有大量临时工具调用信息；只用 memory，又会丢失当前会话的精确状态。所以项目把 session 状态和长期记忆分开治理。

## 本地文件和命令执行的安全边界

Agent 有读写文件和执行命令的能力，如果直接暴露真实 Windows 路径，会有误删、覆盖、越权访问的风险。所以项目设计了本地沙箱和虚拟路径映射。Agent 在 prompt 和工具里看到的是 /mnt/user-data/workspace、/mnt/user-data/uploads、/mnt/user-data/outputs，底层再映射到 .deer_flow/threads/{thread_id}/... 这样的真实目录。

## Agent 运行过程需要统一治理，而不是把所有逻辑塞进主流程。

项目用了 Middleware 架构，把很多横切能力拆出去，比如：

ThreadDataMiddleware 注入 thread 级 workspace / uploads / outputs
MemoryMiddleware 在对话结束后异步更新长期记忆
DeferredToolFilterMiddleware 过滤 deferred MCP tool schema
LoopDetectionMiddleware 检测重复工具调用，避免死循环
ClarificationMiddleware 处理需要澄清的任务
ToolErrorHandlingMiddleware 做工具错误兜底
SummarizationMiddleware 做长上下文压缩

# 项目中遇到了什么问题
## MCP 工具 schema 太多，prompt 开销很高。

一开始如果直接把所有 MCP tools 都放进 tools，模型每一轮都要看到完整 schema。随着工具增加，成本和干扰都会上升。我的解决方案是 tool_search + DeferredToolRegistry + DeferredToolFilterMiddleware。

难点在于不能简单把 MCP 工具移除，因为执行层还要能调用它们。所以我把工具分成两层：执行层保留完整工具，模型可见层延迟暴露 schema。型调用 tool_search 后，才把对应工具 promote。

这个方案也有权衡：第一次调用 deferred 工具可能多一轮 tool_search，交互步数增加；搜索质量依赖工具名和 description。所以我支持了 select:name 精确选择、+keyword 强制关键词和普通 regex 搜索。

## 并发请求下 deferred 工具状态不能串线

如果 DeferredToolRegistry 用普通全局变量，A 会话 promote 的工具可能影响 B 会话，导致 B 会话突然看到不该看到的 schema。这个问题在多 Agent、多线程或异步 LangGraph 执行时很危险。

解决方式是用 ContextVar 保存 registry，让每个 graph run / async context 有自己的 deferred tool 状态。这样工具发现、promote、调用链路都只作用于当前请求。

## 长期记忆容易被临时上下文污染

比如用户临时上传一个 PDF，让 Agent 分析。如果系统把“用户上传了某某文件”写进长期记忆，下一次会话 Agent 可能误以为这个文件还存在，甚至去访问已经失效的路径。

所以项目在 MemoryMiddleware 和 memory updater 里做了过滤：只保留用户真实问题和最终 assistant 回复，跳过工具消息、中间 tool call、上传文件块等临时上下文。长期记忆不是日志，而是沉淀后的稳定信息。

## 每轮对话都同步更新 memory 会影响性能和稳定性。

如果每次回答完都立刻调用 LLM 总结并写 memory.json，会阻塞主对话流程，增加延迟和成本；用户连续发消息时，还可能出现多个 memory 更新互相覆盖的问题。

所以项目用了 memory queue 和 debounce。对同一个 thread 的短时间多轮更新会合并，只保留最新上下文，再异步更新 memory。这样主流程响应更快，记忆写入也更稳定。

## Agent 容易重复调用工具形成循环

比如模型连续多次调用同一个失败工具，或者用同样参数不断重试。这会浪费 token 和时间，也可能卡住用户请求。所以项目加了 loop detection middleware，通过检测重复 tool-call pattern，在达到阈值时给 warning 或 hard stop，让 Agent 停下来重新规划
