# 中间件

它们挂在 LangChain/LangGraph agent 的生命周期钩子上，控制模型调用、工具调用、上下文、状态、沙箱和记忆。

## 设计理念
这个项目的中间件设计核心是：**把 agent 的不可控行为拆成可治理的运行时层。**
README 里也写得很明确：项目关注的是多 agent 本地运行时问题，包括上下文控制、工具治理、记忆持久化、会话隔离、本地沙箱、MCP 集成、自定义 agent 行为。见 README.md (line 9)。
所以这些中间件不是随便加的，它们分别对应几个实际痛点：

```
ThreadData + Sandbox 解决隔离问题：每个会话有自己的目录和执行环境。

Uploads + Summarization + Memory 解决上下文问题：文件不全塞上下文，长对话可摘要，有价值内容进入长期记忆。

ToolError + LLMError + DanglingToolCall 解决稳定性问题：工具、模型、历史消息格式任一环出错，都尽量不中断整个对话。

SandboxAudit + DeferredToolFilter + tool_search 解决工具面过大的安全和成本问题：bash 要审计，MCP 工具要延迟暴露，模型不该默认看到所有能力。

Todo + LoopDetection + Clarification 解决 agent 行为质量问题：复杂任务要持续推进，死循环要打断，不确定需求要停下来问人。
```

## 整体组成以及作用
```md
1, 运行环境类
ThreadDataMiddleware：为每个 thread_id 准备独立目录结构，包括 workspace、uploads、outputs。作用是会话隔离，避免不同对话互相污染。见 thread_data_middleware.py (line 23)。

SandboxMiddleware：为 thread 分配沙箱环境，工具首次调用时懒加载，复用同一会话沙箱。作用是把文件、命令执行和会话状态绑定起来。见 sandbox/middleware.py (line 21)。

UploadsMiddleware：把用户上传文件的信息注入最后一条 human message，包括文件路径、大小、文档 outline/preview。作用是让模型知道“有哪些文件可读”，但不直接把完整文件塞进上下文。见 uploads_middleware.py (line 65)。

2, 模型调用稳定性类
DanglingToolCallMiddleware：修复“AIMessage 有 tool_calls，但历史里没有对应 ToolMessage”的坏消息链。常见于中断、取消、恢复会话后，否则模型 API 会报格式错误。见 dangling_tool_call_middleware.py (line 29)。

LLMErrorHandlingMiddleware：模型请求失败时做重试、退避，并区分 quota/auth/transient/busy 等错误，最后转成用户可读的 AIMessage。见 llm_error_handling_middleware.py (line 68)。

SummarizationMiddleware：来自 LangChain 官方中间件，由 _create_summarization_middleware() 按配置创建，用于长上下文压缩，减少 token 压力。配置在 config.yaml 的 summarization 段。

3, 工具治理类
ToolErrorHandlingMiddleware：工具异常不会直接炸掉整次 agent run，而是转换成 error ToolMessage，让模型可以继续解释失败原因或改用其他方案。见 tool_error_handling_middleware.py (line 18)。

SandboxAuditMiddleware：专门审计 bash 工具调用，按规则分为 pass/warn/block。高风险命令直接阻断，中风险命令执行但附加警告。见 sandbox_audit_middleware.py (line 203)。

DeferredToolFilterMiddleware：配合 tool_search 使用。MCP 工具很多时，不把所有 schema 直接绑定给模型，而是先隐藏；模型需要时调用 tool_search 获取 schema，再“提升”为可调用工具。见 deferred_tool_filter_middleware.py (line 26) 和 tool_search.py (line 40)。

4, 任务控制类
TodoMiddleware：继承 LangChain 的 TodoListMiddleware，增强了两个点：上下文被摘要后提醒模型仍有 todo；模型想提前结束但 todo 未完成时，把它拉回模型节点继续执行。见 todo_middleware.py (line 65)。

LoopDetectionMiddleware：检测重复工具调用循环。相同工具+参数重复到阈值后先警告，再硬停止工具调用，强制模型输出最终答复。见 loop_detection_middleware.py (line 147)。

5, 用户交互类
ClarificationMiddleware：拦截 ask_clarification 工具调用，把澄清问题格式化后用 Command(goto=END) 中断执行，交给前端/用户响应。它被放在最后，确保其他工具治理先发生。见 clarification_middleware.py (line 21)。

TitleMiddleware：首轮完整对话后生成会话标题，失败时本地 fallback。见 title_middleware.py (line 22)。

6, 记忆类
MemoryMiddleware：agent 执行后，把“用户输入 + 最终 assistant 响应”排入记忆更新队列，过滤掉工具消息、上传文件块、中间 tool call。还会检测用户纠正/正反馈。见 memory_middleware.py (line 183)。
```

# TodoMiddleware
作用可以理解成：给 agent 加一个“计划/待办状态机”。它让模型能用 write_todos 工具维护 state["todos"]，并额外防止两类问题：上下文被摘要后忘记 todo，以及 todo 没做完就提前最终回复。

继承自原版 TodoListMiddleware 后，它天然获得三件事：
```
1. 给 agent 注册 write_todos 工具
原版在 .venv/.../todo.py:186 动态创建工具，并在 .venv/.../todo.py:199 放到 self.tools。

2. 给模型注入 todo 使用说明
原版 wrap_model_call 会在每次模型调用前，把 system_prompt 拼进 system message，见 .venv/.../todo.py:201。

3. 在 state 里维护 todos
你的 ThreadState 也声明了 todos 字段：thread_state.py (line 31)。
```

## 输入输出
```write_todos``` 工具输入大概是：

```python
[
    {"content": "分析需求", "status": "completed"},
    {"content": "修改代码", "status": "in_progress"},
    {"content": "运行测试", "status": "pending"},
]

每个ToDo有两个核心字段：
content: str
status: "pending" | "in_progress" | "completed"

```

工具输出不是普通文本，而是 LangGraph Command，会更新 agent state：
```python
{
    "todos": todos,
    "messages": [ToolMessage("Updated todo list to ...")]
}
```
## 自定义类
继承了langchain的TodoListMiddleware类，

```before_model``` 防止“上下文摘要后忘记 todo”
```
class TodoMiddleware(TodoListMiddleware):

会进行判断：1，todo是否再messages中；2，reminder是否再messages中；3，如果都不在的话，就会注入一个提醒：HumanMessage(name="todo_reminder")，提醒模型：“todo 还在，继续维护它。”
```

```after_model``` 防止“todo 没完成就最终回复”。
```python

# 先保留原版逻辑：禁止同一轮并行调用多个 write_todos。
base_result = super().after_model(state, runtime)

# 只有当模型没有工具调用、看起来准备结束时，才介入。
# 找到最后一条AIMessage，
last_ai = 最后一条 AIMessage
if not last_ai or last_ai.tool_calls:
    return None

# 找todo，如果都完成了就结束
todos = state.get("todos") or []
if not todos or all(t["status"] == "completed" for t in todos):
    return None

# 否则就进入限制
return {
    "jump_to": "model",
    "messages": [HumanMessage(name="todo_completion_reminder", ...)]
}
```


# ClarificationMiddleware 
澄清有两个东西：```ask_clarification_tool```和```ClarificationMiddleware```

ask_clarification_tools是BUILTIN_TOOLS，他的schema和description会交给llm；所以llm就可以调用ask_clarification_tools这个工具

在ClarificationMiddleware中，执行流程是```wrap_tool_call```, 会检查：
```python
if request.tool_call.get("name") != "ask_clarification":
    # Not a clarification call, execute normally
    return await handler(request)

return self._handle_clarification(request)
```

## 输入

wrap_tool_call 的输入是 ToolCallRequest，里面关键是:

```python
request.tool_call = {
    "name": "ask_clarification",
    "id": "...",
    "args": {
        "question": "...",
        "clarification_type": "missing_info",
        "context": "...",
        "options": [...]
    }
}
```
_handle_clarification 会读取：
```
args = request.tool_call.get("args", {})
question = args.get("question", "")
tool_call_id = request.tool_call.get("id", "")
```
然后进行格式化，最后直接返还给用户：
```python

ToolMessage(
    content=formatted_message,
    tool_call_id=tool_call_id,
    name="ask_clarification",
)

Command(
    update={"messages": [tool_message]},
    goto=END,
)
```
